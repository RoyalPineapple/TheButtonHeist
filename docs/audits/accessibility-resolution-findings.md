# Accessibility resolution layer audit

Scope: TheStash (registry, resolution, wire conversion, diagnostics), TheBurglar (parse/apply pipeline), AccessibilityHierarchy traversal, trait mapping (UIKit + AccessibilitySnapshotBH parser), container types.
Date: 2026-05-11.
Audit-only — no source changes.

Note on naming: the audit brief refers to "TheBagman" for the registry layer, but the actual crew member is `TheStash` (file `ButtonHeist/Sources/TheInsideJob/TheStash/`). TheSafecracker is the touch-injection layer, not element resolution. Findings below use the real names.

## Summary

Ten findings: 1 critical, 6 recommended, 3 optional. The critical finding is a duplicate `(.secureTextField, "secureTextField")` entry in the upstream parser's `knownTraits` table — every secure text field on the wire carries `traits: ["secureTextField", "secureTextField"]`, and the existing `Set`-based sync test cannot catch the duplication. Dominant themes: (1) the trait-mapping table in the AccessibilitySnapshotBH fork now interleaves two private-trait sets and the merge created a duplicate plus orphaned-from-priority traits, and (2) several declared philosophies (memory: "exact or miss — no fuzzy resolution"; README: "matching is exact or miss") have quietly drifted out of sync with the implementation (exact-then-substring two-pass) without updating the docs or the memory file. Highest-leverage fixes: (a) fix the `secureTextField` duplicate; (b) clip `traitNames` against a `Set` at the parser boundary so any future duplicate entry can't leak; (c) make `HeistElement.matches` (client-side filter) and `AccessibilityElement.matches` (server-side resolution) agree about typography normalization and exact-vs-substring semantics — today they don't.

## Findings

### Finding 1: Duplicate `secureTextField` entry in parser `knownTraits` table
**Severity:** critical
**Area:** 4 (traits)
**Location:** `AccessibilitySnapshotBH/Sources/AccessibilitySnapshot/Parser/Swift/Classes/AccessibilityHierarchy+Codable.swift:37` and `:48` (parser submodule, version `f6093b7`).
**Description:** `(.secureTextField, "secureTextField")` appears twice in the `knownTraits` array — once in the "Private traits — core set" block (line 37) and again in the "Private traits — extended set" block (line 48). `traitNames` is computed as `Self.knownTraits.compactMap { contains($0.trait) ? $0.name : nil }`, which iterates the array and emits the name once per matching entry. Any element with `.secureTextField` in its bitmask produces `["secureTextField", "secureTextField"]`. That string array is converted to `[HeistTrait]` via `WireConversion.traitNames` (`ButtonHeist/Sources/TheInsideJob/TheStash/WireConversion.swift:64-66`) without deduplication, so `HeistElement.traits` ships duplicated to every client (CLI compact view, MCP JSON, heist file recordings). Token waste, confusing UX. Property-level deltas (`WireConversion.buildElementUpdate`, `:612-616`) compare `old.traits != new.traits`; the duplication is stable across snapshots so deltas don't churn, but a future server that fixes the duplication would emit a spurious `traits` property change for every secure field after a client upgrade.
**Recommended fix:** Delete line 48 of the parser file. Separately, dedupe at the boundary: in `WireConversion.traitNames`, materialize through `Array(Set(...))` (or wrap parser output in a `Set` before mapping) so any future duplicate table entry can't reach the wire. Two-layer defense.
**Test coverage:** None. `testHeistTraitAllCasesMatchParser` (`ButtonHeist/Tests/TheInsideJobTests/WireConversionTests.swift:134-139`) compares `Set(HeistTrait.allCases.rawValues)` against `knownTraitNames` (`Set<String>`) — Set semantics hide the duplicate. A regression test would assert `traits.traitNames.count == Set(traits.traitNames).count` for every single-trait bitmask in `HeistTrait.allCases`.

### Finding 2: `findElement(heistId:)` resolves to a stale orphan when developer identifiers collide across screens
**Severity:** recommended
**Area:** 1 (heistId)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/ElementRegistry.swift:119-141` (`resolveHeistId`).
**Description:** When `IdAssignment.assign` returns a base heistId equal to an existing registry entry whose identity (label / id / stable-trait set) differs, `resolveHeistId` falls through both branches and returns `baseHeistId` unchanged whenever (a) the existing element has no `contentSpaceOrigin` (non-scrollable case) OR (b) origins are within 0.5pt. The merge then ends up with two leaves in the persistent tree sharing the same heistId — the old orphan and the freshly parsed element. `validateInvariants` (`ElementRegistry+Merge.swift:489-516`) does flag this as "Duplicate heistId" but is invoked only from tests. Realistic trigger: developer reuses `accessibilityIdentifier = "submit"` across two unrelated screens that briefly coexist during a transition; the first orphans before `clearScreen()` fires; the second registers; both share `"submit"`. Result: `findElement(heistId: "submit")` resolves via `elementByHeistId[heistId]` (the path index, which `buildIndex` overwrites with the LAST encountered path during depth-first walk), so the agent may activate the wrong instance, and the orphan remains visible to `selectElements()` / wire tree until the next prune.
**Recommended fix:** Either (a) in `resolveHeistId`, when an existing entry's identity differs (`!hasSameMinimumMatcher`) but origin disambiguation is unavailable, append a deterministic suffix derived from the new element's signature (e.g. a short hash of label+identifier+traits) rather than colliding; or (b) have `merge` call `validateInvariants` in `#if DEBUG` and assert, so test runs surface the collision deterministically. Option (a) is the durable fix; (b) catches future regressions cheaply.
**Test coverage:** None. `ElementRegistryTreeTests` covers same-origin same-identity (no collision) and different-origin same-base (suffix added), but not different-identity same-base. A test would: register an orphaned element with heistId `"submit"`, then call `register(...)` with a different `AccessibilityElement` that synthesizes the same `"submit"` baseId at the same origin, then assert either no collision in `elementByHeistId` or a distinct resolved id.

### Finding 3: `traits.traitNames` does not iterate a deduplicated source — duplicate-table-entry leak is structural
**Severity:** recommended
**Area:** 4 (traits)
**Location:** `AccessibilitySnapshotBH/Sources/AccessibilitySnapshot/Parser/Swift/Classes/AccessibilityHierarchy+Codable.swift:70-72`.
**Description:** `var traitNames: [String] { Self.knownTraits.compactMap { contains($0.trait) ? $0.name : nil } }`. This emits one element per matching `(trait, name)` tuple in the array. Because the source-of-truth is the array (not the deduplicated `knownTraitNames: Set<String>`), any duplicate tuple — past or future — leaks duplicates through to every downstream consumer that respects ordering (HeistElement.traits, the agent diagnostic ("[\(traitNames.joined)]"), heistId synthesis priority via `traitPriority.first { contains(mask) }`). Finding 1 is the present instance; this finding is the structural risk that creates it.
**Recommended fix:** Either implement as `var traitNames: [String] { Self.knownTraits.reduce(into: []) { acc, pair in if contains(pair.trait), !acc.contains(pair.name) { acc.append(pair.name) } } }` (O(n²), but `n=42`), or simply assert `knownTraits.count == knownTraitNames.count` in a unit test inside the parser package so the build fails if a duplicate is ever added. The unit-test approach is the cheapest defense.
**Test coverage:** Should add a parser-package test `XCTAssertEqual(UIAccessibilityTraits.knownTraits.count, UIAccessibilityTraits.knownTraitNames.count)`. Currently absent.

### Finding 4: Resolution-side and client-side `matches(...)` use different semantics for the same `ElementMatcher`
**Severity:** recommended
**Area:** 2 (resolution)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/TheStash+Matching.swift:116-176` (server: `AccessibilityElement.matches`) vs `ButtonHeist/Sources/TheScore/Elements.swift:718-741` (client: `HeistElement.matches`).
**Description:** Server-side resolution runs an exact-then-substring two-pass (`matchScreenElements`, `:218-223`) and folds typographic punctuation (`normalizeTypography`, `:162-205`) before comparison. Client-side `HeistElement.matches` only ever does `localizedCaseInsensitiveContains` (substring) and does NOT fold smart quotes / dashes / ellipsis. Same `ElementMatcher`, two behaviours. Specific divergences: (a) An exact match that wins server-side resolves uniquely; the client-side filter would also include all substring matches and not differentiate. (b) `label="don't"` (ASCII apostrophe) matches a server-side element label `"don\u{2019}t"` (right single quotation mark) but does NOT match the wire-form copy of that element on the client. (c) Empty matcher labels — client respects "" → returns false; server does the same via `matchLabel.isEmpty { return false }`. The first two divergences are user-visible when an agent or test client filters `Interface.elements` locally and expects the same hit set as a server-side `activate`.
**Recommended fix:** Pull `normalizeTypography` and a mode parameter into `HeistElement.matches`, defaulting client filtering to substring (the existing behaviour) but exposing a `mode:` parameter. Alternately, make `HeistElement.matches` documented as "substring-only, no typography folding" and rename to `matchesSubstring` so the asymmetry is unmissable. Either way, update the matcher doc in `Elements.swift:649-656` (currently says "Trait values use the HeistTrait enum… matching logic lives as an extension on AccessibilityHierarchy"; it neglects to say client-side has a separate, less-forgiving copy).
**Test coverage:** None compares server-and-client behaviour on the same `ElementMatcher`. A test would: build a `HeistElement` with smart-quote label, assert server-side `AccessibilityElement.matches` and client-side `HeistElement.matches` either both true or both false for an ASCII-quote matcher.

### Finding 5: Documented "exact or miss" matching philosophy contradicts implemented two-pass fallback
**Severity:** recommended
**Area:** 2 (resolution)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/README.md:14`, `ButtonHeist/Sources/TheScore/Elements.swift:264` (ElementTarget doc says "case-insensitive substring matching"), memory file `feedback_deterministic_matching.md`. Implementation: `TheStash+Matching.swift:218-223`.
**Description:** The README says matching "0 hits = `.notFound`, 2+ = `.ambiguous`" with no mention of exact-vs-substring. The matcher case doc on `ElementTarget` says "Matcher fields use case-insensitive substring matching" — no mention of exact-first. The memory file says "Exact match on heistId/identifier or miss. No fuzzy matching. On exact miss, return substring matches as suggestions ("did you mean X or Y?"). Never auto-resolve to a fuzzy match." But the code in commit `6397f898` (Apr 29, "Match exact-first, fall back to substring") DOES auto-resolve to substring when exact has no hits. Behaviorally reasonable, but every doc surface still describes one of the older semantics. Agents reading the README will build mental models that diverge from production behaviour, and the memory file will mislead future code reviewers.
**Recommended fix:** Pick the truth, then sync. Two options: (1) keep the two-pass and update README, ElementTarget doc, and the memory file to describe "exact pass first; substring fallback for compatibility with imprecise matchers; substring matches are reported as `ambiguous` if they collide"; (2) revert to exact-only and surface substring hits as suggestion strings in `.notFound` diagnostics, matching the memory's stated philosophy. Either is defensible — what isn't defensible is leaving three separate docs disagreeing with the code. The audit raises this as `recommended` rather than `optional` because contract drift is the kind of silent bug that breeds future "regressions".
**Test coverage:** `TheStashResolutionTests` covers both passes' end-state but doesn't assert that exact wins outright when both substring and exact would match. Add a test: register `["Save", "Save Draft"]`, query `label="Save"`, assert `.resolved` to `"Save"` (not `.ambiguous`).

### Finding 6: `findNearMiss` operates only on `currentHierarchy`, skipping off-screen elements in the registry
**Severity:** optional
**Area:** 2 (resolution)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/Diagnostics.swift:80-138`.
**Description:** When a matcher fails to resolve, `findNearMiss` relaxes each predicate in turn and re-runs against `hierarchy` only (`hierarchy.firstMatch(relaxation.relaxed, mode: .substring)`). Off-screen elements that were previously explored and live in the registry (but not the live hierarchy) are not consulted. So an agent that explores a long list, scrolls it out of view, then re-queries with a slight typo gets a generic compact-summary fallback instead of "near miss: matched all fields except label — actual label='Submit Order'". The same diagnostic shape would be much more useful with registry visibility.
**Recommended fix:** Extend `findNearMiss` to walk the registry's `flattenElements()` after the hierarchy probe, applying the same relaxation pipeline. The first match (hierarchy-or-registry) wins; report registry hits with an "(off-screen)" suffix so the agent knows to scroll-to-visible before activating. Modest token cost in the failure path, real win when the agent is exploring large lists.
**Test coverage:** No tests for the registry-fallback near-miss path. A test would register an off-screen element with label "Submit Order" and assert that querying `label="Sumbit Order"` returns a near-miss diagnostic mentioning "Submit Order" rather than a compact summary.

### Finding 7: `traitPriority` for heistId synthesis is missing `tabBarItem`; `header` is ranked below `button`
**Severity:** recommended
**Area:** 1 (heistId)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/IdAssignment.swift:19-30`.
**Description:** `traitPriority` orders `backButton → searchField → textEntry → switchButton → adjustable → button → link → image → header → tabBar`. Two issues. (a) `tabBarItem` (private trait, bit 28) is absent — a tab bar item's heistId synthesizes via `.button` instead of `.tabBarItem` because `tabBarItem` typically co-occurs with `.button` and `.button` is hit first. Tab items get `home_button` rather than `home_tabBarItem`. Functional but ambiguous when a screen has both a "Home" tab item and a "Home" button. (b) `.button` outranks `.header` — so a section header marked `[.header, .button]` (a "tap to expand" header) synthesizes as `*_button`, not `*_header`, even though the header role is the more descriptive trait. The README at `TheStash/README.md:28` claims the order is "backButton → header → tabBar"; the actual order has `header` near the bottom. Doc drift compounds the ranking problem.
**Recommended fix:** Insert `tabBarItem` between `tabBar` and `backButton` (or as the highest-priority private trait after `backButton`). Move `header` above `button` and `link` so semantic role wins over interaction role when both are present. Update the README to match.
**Test coverage:** `IdAssignmentTests` covers individual trait priorities (`testBackButtonTraitTakesPriority`, etc.) but never tests trait combinations. Add tests: `[.header, .button]` → `*_header`; `[.tabBarItem, .button]` → `*_tabBarItem`.

### Finding 8: `wait_for absent` (`hasTarget` matcher path) substring-only — diverges from `resolveTarget` exact-first
**Severity:** optional
**Area:** 2 (resolution)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/TheStash.swift:326-335` (`hasTarget`).
**Description:** `hasTarget(.matcher(...))` always runs `currentHierarchy.hasMatch(matcher, mode: .substring)`. `resolveTarget(.matcher(...))` runs the two-pass exact-first matcher. For most cases this is correct because exact ⊂ substring, so `hasTarget` returns true whenever an exact match would resolve. But: `hasTarget` does NOT use the registry, by design (comment: "wait_for absent checks live hierarchy only"); `resolveTarget` DOES fall back to the registry on hierarchy miss. So `wait_for absent label="X"` can claim "X is absent" (no live hierarchy hit) while `resolveTarget` succeeds against an off-screen registry entry. An agent that polls `wait_for absent` to confirm a screen dismissed could then try to activate the dismissed element and succeed — confusing.
**Recommended fix:** Either (a) document explicitly that `wait_for absent` is "absent from the live screen, not from memory" — this matches the existing comment — and lean on `hasTarget` returning false to mean "scrolled out or dismissed, not necessarily gone". Or (b) split into two predicates: `hasLiveTarget` (current behaviour) and `hasMemoryTarget` (consults registry). Option (a) is the cheapest; (b) is the principled fix.
**Test coverage:** `testHasTargetIgnoresOffScreenMatcherInRegistry` and `testHasTargetIgnoresOffScreenHeistIdInRegistry` already cover this — they assert the divergent behaviour. The audit gap is documentation, not test coverage.

### Finding 9: Unknown trait bits (e.g. AXRuntime bit 33) silently dropped from the wire
**Severity:** optional
**Area:** 4 (traits)
**Location:** `AccessibilitySnapshotBH/Sources/AccessibilitySnapshot/Parser/Swift/Classes/AccessibilityHierarchy+Codable.swift:70-72` (`traitNames`); `ButtonHeist/Sources/TheInsideJob/TheStash/WireConversion.swift:64-66` (`traitNames(_:)`).
**Description:** `traitNames` only iterates `knownTraits`; bits not in the table are dropped entirely from the `[String]` result. Codable encoding of the bitmask handles unknown bits via `"unknown(\(rawValue))"` (lines 109-121), but `WireConversion.traitNames` does not — it reads only the named-traits view. Net effect: if iOS adds a new private trait at, say, bit 41, or if a future patch adds bit 33 to `UIAccessibilityTraits`, every element with that bit gets the bit silently stripped from the wire trait list, even though the bitmask carries it. Per memory `feedback_two_trait_spaces.md`: "XCTest snapshot (attr 2004) has the full AXRuntime superset including hidden bit 33". If the parser ever consumes attr 2004, bit 33 would be dropped here.
**Recommended fix:** Compute the residual rawValue after subtracting all known traits (the same logic in Codable encode at lines 113-118). If non-zero, append a single `.unknown("0x\(hex)")` HeistTrait. Forward-compat parity with the Codable path; tiny payload cost. Alternatively, document explicitly that the wire path strips unknown bits and the agent should fall back to the bitmask for newer iOS versions.
**Test coverage:** None. A test would construct `UIAccessibilityTraits(rawValue: 1 << 33)` and assert that `WireConversion.traitNames` either emits an unknown trait or that there is a documented design reason it doesn't.

### Finding 10: `RegistryNode` invariants drift from production by relying on `validateInvariants` being test-only
**Severity:** optional
**Area:** 1 (heistId) / 3 (traversal)
**Location:** `ButtonHeist/Sources/TheInsideJob/TheStash/ElementRegistry+Merge.swift:480-533`.
**Description:** `validateInvariants` checks four real invariants (path consistency, leaf-index symmetry, no duplicate heistIds, no empty containers). It is invoked exclusively from `ElementRegistryTreeTests`. The merge pipeline is pure, so in principle invariants hold by construction — but the entry points that mutate the registry (`register`, `merge`, `prune`, `clearScreen`, `clear`, `insertForTesting`) are six surfaces and growing; the recent `containersNestedInScrollView` and `scrollableViews` parameters in particular extend the merge contract without any DEBUG-time guard. If Finding 2's collision scenario or a future pipeline change breaks an invariant, production code silently produces a malformed tree and `findElement` / `selectElements` will return wrong results. Tests cover the cases their authors thought of, not the ones they didn't.
**Recommended fix:** Call `validateInvariants` from the end of `merge`, gated on `#if DEBUG` (which the whole module is anyway), and `assertionFailure` on a violation. Zero release cost; surfaces every test-build invariant break at the merge site rather than waiting for a downstream `findElement` to act on a bad path.
**Test coverage:** Existing tests cover every invariant individually. The gap is enforcement at production-debug-time, not test-time.

## What was checked and NOT found

**Area 1 (heistId)** — Verified: `IdAssignment.assign` is pure value-in/value-out (`IdAssignment.swift:38-62`); two-phase base-id-then-suffix disambiguation handles within-parse duplicates; developer identifiers take priority including the UUID-skip rule (`isStableIdentifier`, `Elements.swift:358-361`); content-space suffix `_at_X_Y` disambiguates scroll-repeated elements (`ElementRegistry.swift:143-145`); transient traits (`selected`, `notEnabled`, `isEditing`, `inactive`, `visited`, `updatesFrequently`) are excluded from `hasSameMinimumMatcher` so toggling state does not break id stability (`ElementRegistry.swift:167-174`). Did NOT find: any path where heistId derivation reaches into wire types or scroll-view geometry; any persistence of heistIds across app reinstalls (no Keychain or NSUserDefaults references — heistIds are session-scoped, which is fine but should be an explicit product position).

**Area 2 (resolution)** — Verified: `resolveTarget` dispatches symmetrically for `.heistId` (O(1) via `elementByHeistId`) and `.matcher` (`TheStash.swift:266-278`); ambiguity surfaces up to 10 candidates with deterministic ordering (`ambiguousResolution`, `:382-402`); ordinal selection clamps negative ordinals (`:283-286`) and out-of-bounds (`:289-291`); the registry-fallback for off-screen matcher hits is deterministic via heistId ordering for ties (`registryOrder`, `:247-260`). Did NOT find: case-sensitivity drift across the resolution path (everywhere uses `localizedCaseInsensitive*`); leakage of substring matching into the trait predicate path (trait checks are bitmask exact at every level).

**Area 3 (traversal)** — Verified: `folded`/`mapTree` primitives on both `AccessibilityHierarchy` and `RegistryNode` cover every recursive operation in the pipeline; the compactMap-with-context helper supports early-exit via `first:` (`AccessibilityHierarchy+TreeOperations.swift:53-79`), and `matches(limit:)` uses it (`TheStash+Matching.swift:87-96`); no ad-hoc recursive walks remain — `firstHeistId`, `collectOrphans`, `buildIndex`, `attachInside`, `validateContainers` are hand-written but each has unique semantics that `folded` doesn't fit (per the PR-326 review note). Did NOT find: lazy-pipeline gaps — the only `.lazy` use in the resolution layer is `matchScreenElements` registry fallback (`TheStash+Matching.swift:240-244`), and it correctly chains filter+prefix; cross-fork drift in the parser's traversalIndex ordering vs VoiceOver order; recursive container references (the parser builds finite trees from UIKit, no cycle possible).

**Area 4 (traits)** — Verified beyond Findings 1/3/9: `excludeTraits` correctly uses `isDisjoint(with:)` for bitmask comparison (`TheStash+Matching.swift:141`); unknown trait names in a matcher fail-safe to "no match" rather than match-everything (`:130-134, 139-141`); `HeistTrait.unknown(_)` preserves wire-form round-tripping (`Elements.swift:148-159`); the parser's Codable encode/decode handles unknown trait bits via `"unknown(\(rawValue))"` (`AccessibilityHierarchy+Codable.swift:90-121`). Did NOT find: any place where bitmask comparison drifts to string comparison; any test that asserts `traits.traitNames.count == Set(traits.traitNames).count` (the gap that lets Finding 1 exist).

**Area 5 (containers)** — Verified: `.scrollable(contentSize:)` flows from parser (`AccessibilityContainer.swift:23`) → registry (`RegistryContainerEntry`, container preserved) → wire (`ContainerInfo.ContainerType.scrollable(contentWidth:contentHeight:)`, `Elements.swift:376`) → consumer (`TheBrains+Scroll.swift:537-568`, used for swipe-target keys and scroll-axis detection); container `stableId` derivation is deterministic by type with content-frame-coarsening for non-semantic containers (`ElementRegistry+Merge.swift:37-65`); top-level scrollables get `ObjectIdentifier`-based stableId so frame drift doesn't detach retained children; tab-bar vs content partition uses depth-first walk via `compactMap` (`TheBurglar.swift:243-260`). Did NOT find: alert-vs-tree integration — `TheMuscle.swift` does present a UIAlertController via `AlertPresenter` for auth approval, and that alert WOULD appear in the accessibility tree because the parser walks all accessible windows; this is by design (the agent sees the approval prompt as elements). Did NOT find: any container-collision scenarios in the merge pipeline (the `stableId` heuristic is documented as "container identity may collide when frames quantize identically" but that case is handled by the `prune empty containers` step after orphan attachment, not by collision detection per se — narrow edge case, unlikely in practice).

## Open questions

- **heistId stability across app launches.** Today they are session-scoped (no persistence). Heist playback ships `ElementMatcher`-based selectors, not heistIds, so this is intentional. Worth saying so in the API doc explicitly so a future agent doesn't try to persist heistIds in test fixtures.
- **`accessibilityIdentifier` collision across screens.** Should two screens that legitimately reuse the same developer-set identifier on different elements be treated as the same heistId, or be force-disambiguated? Current code returns the bare identifier and risks Finding 2's collision. Product decision: are identifiers meant to be globally unique within the app, or only per-screen?
- **Exact-then-substring vs exact-or-suggestions.** Finding 5 is a doc/code drift but also a real product question. The memory `feedback_deterministic_matching.md` strongly prefers "fail loudly with suggestions" over "auto-resolve to fuzzy match"; the implementation went the other way. Worth confirming which is the desired contract going forward.
- **`tabBarItem` heistId synthesis.** Should tab items get their own trait suffix (Finding 7), or is hiding inside `*_button` an acceptable token-saving compromise?
- **Unknown trait bit forward-compat (Finding 9).** Drop unknown bits silently, surface as `unknown(<hex>)`, or surface as a single `unknown_traits` boolean? Each is defensible; today we drop, which is the most token-conservative but loses information that AccessibilitySnapshotBH's Codable path preserves.
- **`validateInvariants` in production-debug.** TheInsideJob is `#if DEBUG`-only anyway; running invariant checks at every `merge` is cheap. Should they be enforced (Finding 10), or kept test-only to avoid the assert paths skewing benchmark timings?
