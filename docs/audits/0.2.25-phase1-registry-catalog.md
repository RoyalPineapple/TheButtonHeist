# 0.2.25 Phase 1 — ElementRegistry Callsite Catalog

This document is the audit deliverable for Phase 1 of 0.2.25 (see
`.context/audit/0.2.25-eliminate-registry-state.md`). It catalogs every
callsite of `ElementRegistry` and its members, then resolves the four open
architectural questions that gate the implementation phases.

Branch base: `RoyalPineapple/accessibility-bug-findings-bundle4b` (PR #367).
Read this alongside the plan doc — the plan doc describes the destination;
this doc describes how each existing line gets there.

Categories used throughout:

- **STAYS** — keeps the same shape; the per-parse index is a drop-in for the
  existing `elementByHeistId` / `viewportIds` reads.
- **SIMPLIFIES** — same call signature, much simpler body (e.g. tree walk
  collapses to a single dictionary lookup or `Array.values`).
- **DELETES** — orphan attachment, merge, prune, invariant checking; no
  replacement.
- **MIGRATES** — moves out of `ElementRegistry` into `TheStash` directly
  (mostly container metadata used for wire tree formatting).
- **NEEDS ATTENTION** — depends on cross-parse persistence; either it is in
  the 6% case (off-screen heistId resolution that must now miss with a clear
  error) or it touches the persistent tree for diagnostics / explore caching
  and must be re-derived from per-parse state.

## Section 2: Disambiguation Tie-Breaker Decision

When N elements share `synthesizeBaseId(...)` output in a single parse, the
`_1`/`_2` suffix is currently assigned by **position in the input
`parsedElements` array** (`IdAssignment.assign`, line 60–68). That array is
itself produced by `[AccessibilityHierarchy].sortedElements` in
`ArrayHelpers.swift`-adjacent code — which yields DFS through the parser
hierarchy.

**Decision: keep DFS-position ordering. Use the parser hierarchy's DFS as the
authoritative order.**

Rationale:

1. The parser hierarchy is *already* the canonical traversal order — it
   matches what VoiceOver focuses through. Agents synthesize heistIds based
   on labels and traits; the disambiguation suffix should match what they'd
   see in `get_interface` output, which renders hierarchy-first.
2. `frame.y, frame.x` is tempting but fragile: keyboard accessory views,
   pop-up controllers, and SwiftUI animations can briefly shift screen-space
   coordinates without changing semantic order. The parser hierarchy is
   layout-independent.
3. The May-9 evidence (59% guess rate) shows agents guess the base id; they
   rarely guess `_1` vs `_2`. When they do, the predictable rule is "the one
   that appears first in VoiceOver order" — which is DFS, which is what the
   parser already gives us.
4. This is exactly what Bundle 4b's cursor fix in `ElementRegistry+Merge.swift`
   `buildNodes`/`buildNode` was already enforcing (`heistIds` consumed
   positionally by DFS — see lines 218–237 of the merge file, especially the
   block comment "we own the matching ordinal").
5. The disambiguation phase in `IdAssignment.assign` (line 58–68) already
   does exactly this: walks the input array in order and assigns `_1`, `_2`,
   …. **No code change is needed in 0.2.25 for the tie-breaker itself.** The
   only thing that changes is that the heistId array is no longer routed
   through `resolveHeistId` cross-element disambiguation in `ElementRegistry`.

Edge cases:

- **Overlapping frames** (e.g. two buttons stacked in a ZStack with identical
  label + trait): the second gets `_2`. If the visual stacking order flips,
  the suffix flips. Acceptable — the agent's `_1` guess will hit a different
  element, which is the same outcome it gets today when the registry
  remembers a previous identical pair.
- **Elements at `frame.zero`** (off-screen, recently dequeued, etc.): they
  appear in the parser hierarchy if accessibility-traversable, so they get a
  DFS index alongside on-screen siblings. No special case needed.
- **Identical content elsewhere in the tree** (e.g. "OK" button in a sheet
  modal sibling of "OK" button on the host screen): DFS visits them in the
  hierarchy's stated order. The modal is rendered first in
  `AccessibilityHierarchy` (it's the topmost interactive layer), so its "OK"
  is `_1`. This is the same answer a VoiceOver user gets.

## Section 3: `_at_X_Y` Content-Position Suffix

Today `resolveHeistId` appends `_at_<x>_<y>` when an existing registry
element shares a base heistId but lives at a different `contentSpaceOrigin`.
This is the scroll-aware identifier for elements inside a long list — the
same base "song_a_staticText" becomes "song_a_staticText_at_0_200" when
scrolled past the original.

**Decision: drop `_at_X_Y` entirely in 0.2.25.**

Rationale:

1. Inside a single parse, the parser produces at most one
   `AccessibilityElement` per visible row — re-used cells produce different
   `AccessibilityElement` values (different `description`, `label`, etc.).
   Two truly identical elements at different `contentSpaceOrigin`s would
   only collide if a list shows duplicate rows verbatim, which is rare and
   already covered by the `_1`/`_2` DFS suffix.
2. The `_at_X_Y` suffix was needed *because the registry was persistent*:
   the same element at two scroll positions across parses had to be
   distinguished. Without persistence, this case disappears — the next parse
   only describes what's currently visible.
3. Keeping `_at_X_Y` would make heistIds *less* predictable. Agents that
   guess `song_a_staticText` won't know which scroll position to encode and
   will mis-guess most of the time. Removing the suffix lets the 59% guess
   rate hit consistently.
4. Heist recording playback uses `ElementMatcher`, not heistId — see
   `HeistEvidence` in `TheScore/HeistPlayback.swift`. Dropping `_at_X_Y`
   doesn't break replays.
5. The disambiguation suffix (`_1`, `_2`) already handles duplicate-content
   collisions inside a parse, which is the only remaining collision shape.

Migration note: `ElementRegistry.contentPositionHeistId` deletes;
`signatureHeistId` (`_sig_<hash>`) deletes (it only existed to disambiguate
the "different identity, same base id" case across parses).

## Section 4: Off-Screen Activation Failure Mode

The 6% case. An agent caches a heistId from `get_interface` round N,
navigates / scrolls, and tries to act on it without refetching.

**Today:** the registry retains the element from round N. The agent's
`activate(heistId:)` succeeds with a one-shot jump-to-recorded-position via
`offViewportRegistryEntry` + `jumpToRecordedPosition`.

**After 0.2.25:** the registry has no memory. `resolveTarget(.heistId(id))`
returns `.notFound`. The agent should refetch or scroll first.

### Error message draft

The existing `Diagnostics.heistIdNotFound` is the right entry point. Updated
copy:

```
Element not found: "song_a_staticText" (12 elements on screen)
This heistId is not in the current parse. The element may have scrolled
off-screen, been replaced by a navigation, or never existed.
similar: song_a_staticText, song_b_staticText
Recovery:
  • Call get_interface to refresh — if the element is there, retry.
  • If it is not, call element_search with a matcher to scroll it
    into view.
```

### Diagnostic information available without state

- `index.keys` — full list of current-parse heistIds (already used for
  "similar" suggestions).
- `viewportIds.count` — number of visible elements (kept for the
  `(N elements on screen)` line; in the new model viewportIds == index.keys
  so this is just the count of currently-parsed elements).

### Information NOT available without state

- "Last seen N parses ago" — we have no parse history.
- "Element existed in screen X but not Y" — no cross-screen state.
- "Element was scrolled out of view" — we can't distinguish "scrolled out"
  from "navigated away" without history.

This is fine: the agent does not need a forensic explanation, it needs a
*recovery action*. The error tells it to call `get_interface` or
`element_search`. Both are cheap. The cost of a mis-cached heistId is one
extra round-trip.

### Recovery instruction

The recovery instruction is the most important part of this error. The
agent guide (`docs/AGENT-GUIDE.md`) and MCP guide
(`docs/MCP-AGENT-GUIDE.md`) should call this pattern out explicitly. Draft
phrasing:

> heistIds are valid only within the response that produced them. If an
> earlier response gave you `foo_button` but the element has scrolled out
> of view or the screen has changed, the next `activate` call will return
> "Element not found." Always call `get_interface` (or any action that
> returns a delta) before reusing an old heistId across a navigation.

## Section 5: Heist File Replay Determinism

Audit verdict: **0.2.25 does not affect heist replay.**

Evidence:

1. `HeistEvidence` in `TheScore/HeistPlayback.swift` (lines 44–73) stores
   commands targeted by `ElementMatcher`, not by heistId. Coding keys
   include `label, identifier, value, traits, excludeTraits, ordinal` —
   heistId is absent.
2. `HeistPlayback.currentVersion = 1`; no schema migration is required if
   matcher semantics are unchanged. PR #366 locked matcher semantics
   (exact-or-miss across server and client) and the recorder/player both
   use that.
3. The CLI/MCP recording surface (TheFence record/play handlers) produces
   matcher fields from the input request, not from the resolved element's
   heistId. (Verified by inspection of the codable surface in
   `HeistEvidence`.)
4. `IdAssignment.synthesizeBaseId` is deterministic from element content
   alone — same `(label, trait, identifier, value)` inputs produce the same
   base heistId. Verified by reading the function (lines 73–90).
5. The disambiguation suffix (Section 2) is deterministic from DFS position
   in the parser hierarchy, which is itself deterministic for the same
   input.

The only conceivable replay break would be if a recording somehow carried a
raw heistId in a field that gets re-asserted at replay (e.g. an expectation
that asserts a specific heistId is present). Spot-checked
`ActionExpectation` in `TheScore/ActionExpectation.swift` — expectations
target by matcher fields or element counts, not heistId. No issue.

No migration is required. No recording format change. The
`HeistPlayback.currentVersion` constant stays at 1.

## Appendix: Open Questions

These came up during the audit. They are not blocking Phase 1 but should
be resolved before Phase 3.

1. **Container `stableId` lifetime.** Today `stableId(for:contentFrame:…)`
   computes a hash that is "stable across parses" for the same container —
   e.g. a top-level scrollable identified by `ObjectIdentifier` of its
   `UIScrollView`. After 0.2.25 there is no cross-parse usage of stableId —
   it only feeds the current-parse wire tree. Two options: (a) keep the
   sophisticated stableId so wire tree consumers can still compare container
   identity across responses; (b) downgrade to a frame-only hash since
   nothing inside the iOS side compares them anymore. Recommend (a) — the
   wire tree consumer (the agent) does benefit from container identity
   stability. No code change needed; just leave it.

2. **`ensureOnScreen`'s jump-to-recorded-position.** `TheBrains+Scroll.swift`
   line 605: `ensureOnScreen` jumps a scroll view to a recorded position
   before performing an action, if the element is off-viewport. After
   0.2.25, `offViewportRegistryEntry` will only succeed for matcher targets
   that happen to be in the current parse (which by definition means
   on-screen), making the jump branch dead. The whole optimization deletes.
   Confirm this is acceptable — it means actions targeting `.matcher` for an
   off-screen element will fail with "not found" instead of silently
   scrolling first. The matcher path should call `scroll_to_visible` or
   `element_search` explicitly. **This is a behavior change worth calling
   out in release notes.**

3. **`TheBrains+Exploration` accumulator.** `recordDuringExplore` accumulates
   heistIds across multiple parses in `explorePhase = .active(seen: …)`. The
   accumulator is *itself* cross-parse state. Today it's used to feed
   `registry.prune(keeping: seen)` at the end of the cycle. After 0.2.25
   the prune call deletes — but does the accumulator still serve a purpose?
   It does: `containerExploreStates` caches "I've explored container X and
   it had heistIds Y" so re-explores can skip unchanged containers. So the
   accumulator stays. Just the prune call goes.

## Open Question for Human Input

**Should `ensureOnScreen`'s recorded-position jump (Open Question #2) survive
in any form?** The full delete is the cleanest 0.2.25, but it is a real UX
regression for the matcher-target-off-screen pattern. The plan doc says
"agent must scroll the target into view before activating" — does that
extend to matcher targets, or only heistId targets? If the former: delete
the jump branch entirely. If the latter: keep the matcher branch of
`offViewportRegistryEntry`, but it will now always return nil because
matchers only see the live hierarchy — so the branch is still dead, just
less obviously so. Resolving this requires a UX call from the human owner
about how aggressive 0.2.25 should be about "no scroll memory."
